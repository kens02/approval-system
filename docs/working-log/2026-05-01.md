## タイムライン

### 1. 要件定義書 v1.0 作成

ユーザーから提示された初期要件をもとに、Markdown 形式の要件定義書を作成。

**主要要件**:
- 汎用承認決裁システム（休暇申請・指導受け・その他）
- 申請 → 合議 → 承認 → 決裁 のワークフロー
- 差し戻し・取り消し・メール自動通知
- 初期リリースは休暇申請のみ（年次有給休暇）
- 6名以上の承認者、最低3名
- 休暇単位: 15分時間取得 / 1日取得（1日=465分）
- 月次付与2日（可変）、繰越上限30日（可変）
- 期間: 年度（4-3月）または暦年
- 統計ダッシュボード、印刷出力対応
- 技術: C# ASP.NET 7 + Next.js + 日本語UI

**成果物**: `要件定義書.md`

### 4. 要件定義書 v1.2 へ更新（マルチテナント対応）

ユーザーから追加された3件の重要仕様を反映。

| 追加仕様 | 内容 |
|---|---|
| マルチテナント化 | 全エンティティに `tenant_id` を付与、テナント間データ完全分離 |
| ロール分離 | 「システム管理者（プラットフォーム全体）」と「時間管理者（自テナント全件アクセス・代行決裁・クローズ処理）」 |
| クローズ処理ステップ | 決裁完了（`decided`）→ 時間管理者によるクローズ → `closed` という最終フェーズ追加 |

**ステータス遷移**:
```
draft → in_progress → decided → closed
                  ↘ rejected (soft-delete) + 新申請を draft で生成
                  ↘ cancelled (soft-delete)
```

**コミット**: `8001b77 feat: 要件定義書 v1.2 - マルチテナント対応・時間管理者ロール追加`

### 6. .gitignore の整備（PR #2）

開発中、リポジトリ直下に NVM の痕跡である `$HOME/.nvm/` ディレクトリが
誤って作成されていたことが判明。今後の .NET / Next.js プロジェクト追加に
備え、本格的な `.gitignore` を整備した。

**主な除外対象**:
- `$HOME/`, `.nvm/`（誤作成痕跡）
- .NET: `bin/`, `obj/`, `*.user`, NuGet, EF Core ローカルDB, Publish
- Next.js: `node_modules/`, `.next/`, `.turbo/`, `.swc/`, Playwright
- **シークレット**: `.env*`, `secrets.json`, `appsettings.*.json`, `*.pem`, `*.pfx`
- DB ローカル: `*.sqlite`, `*.db`, `mysql-data/`
- IDE: `.vs/`, `.idea/`, `.vscode/*`（共有推奨設定のみ許可）
- Docker/IaC: `volumes/`, `.terraform/`, `*.tfstate*`

**ブランチ**: `chore/gitignore`

## 主な技術的決定事項

| 項目 | 決定 | 根拠 |
|---|---|---|
| マルチテナント方式 | シングルDB + `tenant_id` カラム + EF Global Query Filter | 集計容易性とコストのバランス |
| 認証 | 独自JWT（HS256 / 60分）+ `IAuthProvider` 抽象化 | 将来の AD/LDAP 連携に備える |
| DB 切替 | `appsettings.json` の `DatabaseProvider` キーで InMemory ⇔ MySQL | 開発の俊敏性と本番性能の両立 |
| 差し戻し処理 | 元申請をソフトデリート + 内容複製した新申請（`parent_application_id` で連結） | 完全な監査証跡を確保しつつ申請者にクリーンな再申請体験を提供 |
| ステータス | `draft / in_progress / decided / closed / rejected / cancelled` の6段階 | クローズ処理を独立した最終フェーズとする |
| 申請番号 | `yyyy-mm-NNNN`（テナント内月次連番） | 可読性と運用上の慣習を尊重 |
| メール送信 | MailKit + `Channel<EmailJob>` + Hosted Service による非同期キュー | API レスポンス性能を維持 |
