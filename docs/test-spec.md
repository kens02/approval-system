## 1. テスト方針

### 1.1 テストレベル

| レベル | 対象 | フレームワーク | 環境 |
|---|---|---|---|
| 単体テスト | ドメイン・ユースケース・ヘルパ | xUnit + FluentAssertions + Moq | InMemory |
| 統合テスト | API エンドポイント | xUnit + WebApplicationFactory | InMemory |
| DB 統合テスト | EF Core / マイグレーション | xUnit + Testcontainers (MySQL) | Docker MySQL |
| E2E | UI シナリオ | Playwright | API + Web 起動 |
| 受入テスト | ユーザーシナリオ | 手動 + Playwright 補助 | staging |

### 1.2 カバレッジ目標

| レベル | 目標 |
|---|---|
| 単体（Domain / Application） | 90% 以上 |
| 統合（API） | 主要シナリオ網羅（200/4xx 系） |
| E2E | 6 大シナリオ（後述）+ 異常系 |

### 1.3 テストデータ

`tests/TestDataBuilder` で Fluent ビルダーを提供:
```csharp
var app = ApplicationBuilder.New()
    .ForTenant(tenantId)
    .ByApplicant(userId)
    .WithLeave(LeaveType.AnnualPaidLeave, takeUnit: TakeUnit.Hour, "2026-05-10 09:00", "2026-05-10 11:00")
    .Build();
```

## 4. E2E シナリオ（Playwright）

### 4.1 シナリオE-1: 標準的な休暇申請フロー

```
1. 申請者でログイン
2. ダッシュボード → 「新規休暇申請」
3. 時間取得を選択し 2026-05-10 09:00-11:00 で申請
4. 残日数表示が 18.00 → 申請内容が反映
5. 「提出」をクリック
6. 申請番号 2026-05-NNNN が表示される
7. 承認者でログインしなおす
8. 承認待ち一覧に表示される → 詳細へ
9. 「承認」をクリック → 次担当者へ進む
10. 決裁者でログイン → 決裁
11. 時間管理者でログイン → クローズ待ち一覧 → クローズ
12. 申請者でログイン → ステータスが「完了」になっている
```

### 4.2 シナリオE-2: 差し戻し

```
1. 申請者が提出
2. 承認者が「差し戻し」+ コメント「理由を明確に」
3. 申請者の通知メールに新申請URLが届く
4. 元申請は履歴一覧で「差し戻し済み」表示
5. 新申請は draft 状態で内容が複製済み
6. 申請者が理由を修正して再提出
7. ステップ1から審査が再開される
```

### 4.3 シナリオE-3: 取り消し

```
1. 申請者が提出
2. ステップ1承認後、申請者が「取り消し」
3. 関与済み担当者全員と時間管理者にメール通知
4. 一覧では「取り消し済み」として表示（フィルタで非表示も可）
```

### 4.4 シナリオE-4: 代行決裁

```
1. 承認者が長期不在
2. 時間管理者が当該申請を開く
3. 「代行承認」ボタンが表示される
4. 代行承認実施 → ログに is_proxy=true で記録
5. 通常通り次ステップへ進む
```

### 4.5 シナリオE-5: マルチテナント分離

```
1. テナントA の管理者がユーザー一覧を開く
2. テナントB のユーザーは表示されないことを確認
3. テナントB の申請ID を直接URLで開く → 404
```

### 4.6 シナリオE-6: 印刷

```
1. 一覧画面で期間=2026年度・印刷を選択
2. プレビュー（A4横）が表示される
3. window.print() を起動
4. 定型用紙印刷も同様に確認
```

## 5. 異常系・境界値テスト

| ID | ケース | 期待 |
|---|---|---|
| EX-001 | 同時 submit を 2 セッションから | 1件成功 / 1件は冪等エラー |
| EX-002 | SMTP 障害時の通知 | 申請は成功、EmailLogs に failed |
| EX-003 | DB 接続断 | 500 + ロールバック |
| EX-004 | 巨大入力（理由 10000 文字） | バリデーションで拒否 |
| EX-005 | XSS 文字列を理由に投入 | エスケープされ表示 |
| EX-006 | 9999件目の翌の申請（NNNN 桁数超過） | 採番失敗もしくは桁拡張 |
| EX-007 | 月跨ぎ深夜 0:00 ちょうどでの提出 | 番号が翌月でリセットされる |
| EX-008 | InMemory → MySQL 切り替え後の起動 | マイグレーション適用 |

## 8. テスト実行・CI

### 8.1 ローカル

```bash
dotnet test                                # 全単体・統合
npm --prefix src/ApprovalSystem.Web test    # フロント単体
npx playwright test                        # E2E
```

### 8.2 GitHub Actions

```
on: pull_request
jobs:
  - dotnet-test (InMemory)
  - dotnet-test-mysql (Testcontainers)
  - web-lint-and-unit
  - e2e (docker compose up → playwright)
```

PRマージ条件: 全ジョブグリーン + カバレッジ閾値クリア。

## 9. 受入テストチェックリスト（抜粋）

- [ ] テナント A・B の管理者が互いのデータを閲覧できないこと
- [ ] 申請→承認→決裁→クローズの一連フローが完了すること
- [ ] 各フェーズで対象者にメールが届くこと（受信者・件名・本文の差し込み）
- [ ] 差し戻し後に新規申請が自動生成され、内容が同一であること
- [ ] 残日数が時間取得後に正しく減算されること
- [ ] 一覧表（A4横）が指定期間で印刷できること
- [ ] 定型用紙が決裁フロー込みで印刷できること
- [ ] システム管理者がテナント停止できること
- [ ] 停止テナントのユーザーがログインできないこと
