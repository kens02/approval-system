## 1. システム概要

### 1.1 目的

各種申請（休暇申請・指導受け・その他）を電子化し、申請から最終決裁・クローズまでのワークフローを一元管理する。**複数組織（テナント）が同一システムを共有して利用できるマルチテナント型**で構築し、組織間のデータは完全に分離する。申請者・合議者・承認者・決裁者・時間管理者が Web ブラウザ上で操作し、各フェーズでのメール自動通知、および決裁済み書類の印刷保存を実現する。

### 1.2 初期リリース範囲

| 機能 | 初期リリース | 拡張 |
|------|:-----------:|:----:|
| マルチテナント（組織分離） | ✅ | - |
| 休暇申請ワークフロー | ✅ | - |
| 時間管理者によるクローズ処理 | ✅ | - |
| 決裁ルート設定（汎用） | ✅ | - |
| メール自動通知 | ✅ | - |
| 印刷・出力 | ✅ | - |
| 統計・ダッシュボード | ✅ | - |
| 指導受け申請 | - | ✅ |
| その他汎用申請 | - | ✅ |

## 2. システム構成

### 2.1 技術スタック

| 層 | 技術 | バージョン | 備考 |
|---|---|---|---|
| フロントエンド | Next.js | 14 以上 | - |
| バックエンド API | C# ASP.NET Core | 7.0 | - |
| ORM | Entity Framework Core | 7.0 | - |
| データベース（開発・検証） | EF Core InMemory | - | ローカル検証用 |
| データベース（本番） | MySQL | 8.0 以上 | 本番切り替え時に接続文字列を変更 |
| メール送信 | SMTP（MailKit 推奨） | - | - |
| 認証 | 独自認証（JWT） | - | 将来 AD/LDAP 連携を考慮した設計 |
| UI コンポーネント | shadcn/ui または Ant Design | - | - |

> **DB 切り替え方針**: `appsettings.json` の `DatabaseProvider` キーで `"InMemory"` / `"MySQL"` を切り替え可能とし、EF Core の `DbContext` 登録部分のみ変更で対応できる構成とする。

### 2.2 システム構成図

```
[ブラウザ (Next.js)]
        ↕ REST API (JSON / JWT)  ← JWT に tenant_id を含めてテナント識別
[ASP.NET Core 7 Web API]
  ├─ テナントミドルウェア（全リクエストで tenant_id を検証・注入）
  └─ ロールベースアクセス制御（RBAC + テナントスコープ）
        ↕
[InMemory DB（開発）/ MySQL 8.0（本番）]
  └─ 全テーブルに tenant_id カラム（テナント間データ分離）
        ↓
[SMTP サーバー（メール通知・非同期）]
```

### 2.3 テナント（組織）概念

- **テナント**: システムを利用する1つの組織単位。全データはテナントで分離される。
- テナントの作成・停止はシステム管理者のみ実行可能。
- 各ユーザーは必ず1つのテナントに所属する（クロステナントは不可）。
- システム管理者はプラットフォーム全体を管理し、テナントには属さない（または専用テナントを持つ）。

### 2.4 ユーザーロール定義

| ロール | スコープ | 説明 |
|--------|--------|------|
| **システム管理者** | プラットフォーム全体 | テナント管理・全テナントのデータ参照。マスタ・設定管理。テナントには所属しない |
| **時間管理者** | 自テナント内 | 自テナントの申請データ全件を閲覧・編集・代行決裁可能。決裁完了後にクローズ処理を行い最終完了させる責任者 |
| **申請者** | 自テナント内 | 申請の作成・提出・取り消し |
| **合議者** | 自テナント内 | リレー順が来た際に意見付与・差し戻しを行う |
| **承認者** | 自テナント内 | リレー順が来た際に承認・差し戻しを行う |
| **決裁者** | 自テナント内 | 最終ステップで決裁・差し戻しを行う |

> 1ユーザーが複数ロールを兼務することができる（例：時間管理者兼決裁者）。  
> システム管理者は特殊ロールであり、テナントロールとは独立する。

## 3. 機能要件

### 3.1 フェーズ管理（ワークフロー）

#### 3.1.1 ワークフロー状態遷移

承認は **リレー方式（1人ずつ順番にメール通知→処理）** で進む。前の担当者が処理を完了した後に次の担当者へ通知される。最終決裁後は **時間管理者によるクローズ処理** を経て「完了」となる。

```
[下書き]
    ↓ 申請者が提出
[ステップ1担当者 へメール通知]
    ↓ ステップ1が承認
[ステップ2担当者 へメール通知]
    ↓ ステップ2が承認
  ・・・（ステップ数だけ繰り返し）
[最終ステップ（決裁者）が決裁]
    ↓
[決裁済み]
    → 申請者・時間管理者へメール通知（※システム管理者への通知は行わない）
    ↓ 時間管理者が内容確認しクローズ操作
[完了（closed）]
    → 申請者へ完了通知メール送信

※ 各承認ステップから:
  ・差し戻し → 現申請IDをソフトデリート（無効化）
              → 申請内容を複製した新規申請IDを自動生成
              → 申請者へ差し戻し通知（差し戻しコメント付き）
              → 申請者が修正・確認して再提出 → ステップ1から再開
  ・取り消し → 申請IDをソフトデリート（取り消し済み）
              → 申請者または時間管理者・システム管理者のみ操作可
```

**申請ステータス定義**

| ステータス | 説明 |
|---|---|
| `draft` | 下書き（未提出） |
| `in_progress` | 承認フロー進行中（現在のステップ番号を別途保持） |
| `decided` | 決裁済み（時間管理者のクローズ待ち） |
| `closed` | **完了**（時間管理者がクローズ処理を実施済み） |
| `rejected` | 差し戻し済み（ソフトデリート対象） |
| `cancelled` | 取り消し済み（ソフトデリート対象） |

#### 3.1.2 ステップの種類

| ステップ種別 | 説明 | 完了条件 |
|---|---|---|
| **合議** | 担当者がコメント・意見を付与する確認ステップ | 担当者1名が処理すると次のステップへ（リレー方式） |
| **承認** | 承認者が内容を確認し承認・差し戻しを行う | 承認で次のステップへ |
| **決裁** | 最終決裁者が正式に可否を決定する | 決裁で `decided` に遷移。申請者・時間管理者へ通知 |
| **クローズ** | 時間管理者が決裁内容を確認し完了とする（承認フローの外の確認処理） | クローズ操作で `closed` に遷移。申請者へ完了通知 |

#### 3.1.3 差し戻し仕様（詳細）

1. ステップ担当者（合議者・承認者・決裁者）が **差し戻し** を実行する
2. 差し戻しコメントは **必須入力**
3. 現在の申請レコードを **ソフトデリート**（`is_deleted = true`、`status = rejected`）
4. 申請内容（休暇詳細）を**そのままコピー**した新規申請レコードを生成
   - 新しい申請番号を採番（`yyyy-mm-連番`）
   - `parent_application_id` に元の申請IDを保持（履歴追跡用）
   - ステータスは `draft` で作成
5. 申請者へ差し戻し通知メールを送信（新申請へのリンク・差し戻しコメントを含む）
6. 申請者が新申請の内容を修正し、再提出するとステップ1から審査を再開する

#### 3.1.4 取り消し仕様（詳細）

- 申請者本人（`in_progress` 状態のみ）または時間管理者・システム管理者が実行可能
- 対象申請レコードを **ソフトデリート**（`is_deleted = true`、`status = cancelled`）
- 関与済みステップ担当者へ取り消し通知メールを送信

#### 3.1.5 クローズ処理仕様（詳細）

1. 最終決裁者が決裁を完了すると申請ステータスが `decided` になる
2. **申請者・テナントの時間管理者全員**へ決裁完了通知メールを送信
3. 時間管理者は通知メールまたはダッシュボードからクローズ待ち一覧を確認する
4. 時間管理者が申請詳細画面で内容を確認し「クローズ」ボタンを押下する
5. ステータスが `closed` に遷移し、`closed_at`・`closed_by_user_id` を記録する
6. 申請者へ「完了」通知メールを送信する
7. 代行決裁: 時間管理者は `in_progress` 状態の申請に対し、現ステップの代わりに承認・差し戻し操作を行うことができる（ログに代行フラグを記録）

#### 3.1.6 ステップ構成の制約

- 決裁ルートに設定できる担当者合計: **最小 3 名 ～ 最大制限なし（推奨 6 名以上）**
- 最低構成例: 申請者（提出） → 承認者（ステップ1） → 決裁者（ステップ2） → 時間管理者（クローズ）
- 各ステップに設定できる担当者は **1名**（リレー方式のため）
- クローズ処理は決裁ルートのステップ設定に依存せず、テナントの時間管理者全員が実行可能

### 3.2 休暇申請機能（初期リリース対象）

#### 3.2.1 休暇取得単位

| 単位 | 説明 |
|---|---|
| **時間取得** | 15 分単位で申請可能 |
| **日取得** | 1 日単位で申請可能（全休・午前・午後を選択） |

- 1 日の労働時間: **7 時間 45 分（465 分）**
- 時間取得と日取得は同一申請内で混在可能とする

#### 3.2.2 休暇残日数の計算ルール

| 項目 | 仕様 |
|---|---|
| 付与サイクル | 月次（毎月 1 日に付与） |
| 月次付与日数 | **2 日**（管理者による変更可） |
| 年度持ち越し上限 | **30 日**（管理者による変更可） |
| 超過分の処置 | 切り捨て |

- 残日数の計算式:  
  `残休暇 = 繰越日数（上限30日） + 当月までの月次付与累計 − 取得済み休暇（時間換算）`
- 時間取得は 465 分（1 日）単位で日数換算して残日数に反映する

#### 3.2.3 休暇申請の入力項目

初期リリースで対応する休暇区分は **年次有給休暇のみ**。他の区分（特別休暇・病気休暇等）は将来の拡張機能として設計上の考慮のみ行う。

| 項目 | 必須 | 説明 |
|---|:-:|---|
| 申請日 | ✅ | 自動セット（編集不可） |
| 申請者氏名 | ✅ | ログインユーザー（自動セット） |
| 休暇区分 | ✅ | **年次有給休暇**（初期リリース固定。将来拡張で他区分追加） |
| 取得単位 | ✅ | 時間取得 / 日取得 |
| 取得日（From） | ✅ | - |
| 取得日（To） | ✅ | 時間取得の場合は From と同日 |
| 取得時刻（From） | 時間取得時 | 15 分単位 |
| 取得時刻（To） | 時間取得時 | 15 分単位 |
| 取得時間数（自動計算） | - | From/To から自動計算 |
| 取得日数（自動計算） | - | 時間取得は合計分数 ÷ 465 分で換算（端数は分単位で保持） |
| 休暇理由 | - | 任意入力 |
| 備考 | - | 任意入力 |

#### 3.2.4 休暇残日数・取得状況

- 申請フォーム上に現在の休暇残日数をリアルタイム表示する
- 残日数を超える申請は警告を表示する（申請自体は可能とする）

### 3.3 決裁ルート設定

#### 3.3.1 決裁ルートの構成

申請者ごとに決裁ルートを設定できる。リレー方式のため各ステップの担当者は **1名**。

| 設定項目 | 説明 |
|---|---|
| 申請者 | ルートを適用するユーザー |
| ルート名 | 管理用の名称 |
| ステップ一覧 | 順序付きのステップリスト（合議・承認・決裁）。順番にリレーされる |
| 各ステップの担当者 | **1名を設定**（リレー方式） |

#### 3.3.2 ユーザーマスタ

| 項目 | 説明 |
|---|---|
| テナントID | 所属テナント（システム管理者は null 可） |
| 氏名（姓・名） | - |
| 表示名 | - |
| メールアドレス | 通知先 |
| 所属部署 | - |
| ロール | システム管理者 / 時間管理者 / 申請者 / 合議者 / 承認者 / 決裁者（複数選択可） |
| 有効/無効フラグ | - |

#### 3.3.3 メール通知文面の設定

決裁ルートのステップごとに通知メールの件名・本文テンプレートを設定できる。  
テンプレートでは以下のプレースホルダーを使用可能とする:

| プレースホルダー | 内容 |
|---|---|
| `{組織名}` | テナントの組織名 |
| `{申請者氏名}` | 申請者のフルネーム |
| `{申請種別}` | 例：休暇申請 |
| `{申請日}` | 申請の作成日 |
| `{取得日From}` | 休暇取得開始日 |
| `{取得日To}` | 休暇取得終了日 |
| `{フェーズ名}` | 現在の処理フェーズ名 |
| `{処理者氏名}` | 前ステップで処理したユーザー名 |
| `{コメント}` | 差し戻しコメントなど |
| `{申請URL}` | 申請詳細画面へのリンク |

### 3.4 通知機能

#### 3.4.1 通知タイミング（リレー方式）

| イベント | 通知先 |
|---|---|
| 申請が提出された | **ステップ1の担当者のみ** |
| ステップN が承認され次へ移行 | **ステップN+1 の担当者のみ** |
| 差し戻しが行われた | **申請者のみ**（差し戻しコメントと新申請URLを含む） |
| 最終ステップ（決裁）が完了した | **申請者 ＋ テナントの時間管理者全員**（承認者・システム管理者への通知は行わない） |
| 時間管理者がクローズ処理を完了した | **申請者のみ**（最終完了通知） |
| 申請が取り消された | 関与済みのステップ担当者全員 ＋ 時間管理者 |

#### 3.4.2 通知メール仕様

- 送信元メールアドレス: システム管理者が設定
- HTML メール / プレーンテキストメール を選択可能
- 送信失敗時はログに記録し、管理画面から再送信できること
- メール送信はバックグラウンドジョブ（非同期）で実行する

## 6. データモデル概要

### 6.1 主要エンティティ

```
【プラットフォーム共通】

Tenants（テナント / 組織マスタ）
  ├── id, name（組織名）, code（組織コード / ユニーク）
  ├── is_active
  └── created_at, updated_at

【テナント共通（全テーブルに tenant_id を付与）】

Users（ユーザーマスタ）
  ├── id, tenant_id（FK: Tenants / システム管理者は null 可）
  ├── name_last, name_first, display_name
  ├── email, department
  ├── roles（カンマ区切り or 別テーブル: system_admin / time_manager /
  │         applicant / consultor / approver / decider）
  ├── password_hash（独自認証用）
  └── is_active, created_at, updated_at

ApprovalRoutes（決裁ルートマスタ）
  ├── id, tenant_id, name, applicant_user_id
  └── created_at, updated_at

ApprovalRouteSteps（ルートステップ）
  ├── id, tenant_id, route_id, step_order（リレー順）
  ├── step_type（consulted / approval / decision）
  └── assignee_user_id（担当者1名 / リレー方式のため単数）

Applications（申請メイン）
  ├── id, tenant_id
  ├── application_number（yyyy-mm-0001 形式 / テナント内ユニーク）
  ├── applicant_user_id, application_type（申請種別）
  ├── status（draft / in_progress / decided / closed / rejected / cancelled）
  ├── current_step_order（現在処理中のステップ番号）
  ├── parent_application_id（差し戻し元の申請ID / NULLは初回申請）
  ├── is_deleted（ソフトデリートフラグ）
  ├── submitted_at, decided_at
  ├── closed_at, closed_by_user_id（クローズ処理日時・実施者）
  └── created_at, updated_at

LeaveApplications（休暇申請詳細）
  ├── id, tenant_id, application_id
  ├── leave_type（休暇区分 / 初期値: annual_paid_leave）
  ├── take_unit（hour / day）
  ├── date_from, date_to, time_from, time_to
  ├── total_minutes（合計分数 / 時間取得・日取得両対応）
  ├── total_days（合計日数換算）
  └── reason, remarks

ApprovalStepLogs（ステップ処理ログ）
  ├── id, tenant_id, application_id, step_order, step_type
  ├── assignee_user_id
  ├── action（approved / rejected / decided / closed / cancelled）
  ├── is_proxy（代行フラグ / 時間管理者が代行操作した場合 true）
  ├── action_at, comment（差し戻し時必須）
  └── is_completed

LeaveBalances（休暇残日数管理）
  ├── id, tenant_id, user_id, balance_year, balance_month
  ├── carry_over_days（繰越日数 / 年度開始月のみ有効値）
  ├── granted_days（当月付与日数）
  ├── used_minutes（申請済み合計分数）
  └── ※ remaining は都度計算（永続化しない）

SystemSettings（システム設定 / テナントごとに1レコード）
  ├── id, tenant_id（ユニーク）
  ├── monthly_grant_days（月次付与日数 / デフォルト: 2）
  ├── max_carryover_days（年度持ち越し上限 / デフォルト: 30）
  ├── work_minutes_per_day（1日労働分数 / デフォルト: 465）
  └── fiscal_year_start_month（年度開始月 / デフォルト: 4）

EmailTemplates（メールテンプレート）
  ├── id, tenant_id
  ├── route_id（NULLの場合はテナント共通テンプレート）
  ├── event_type（submitted / step_approved / rejected / decided / closed / cancelled）
  ├── subject_template, body_template
  └── is_html

EmailLogs（メール送信ログ）
  ├── id, tenant_id, application_id, event_type
  ├── to_email, subject
  ├── sent_at, status（success / failed）
  └── error_message
```

### 6.2 申請番号の採番ルール

- フォーマット: `yyyy-mm-NNNN`
  - `yyyy`: 申請提出年（西暦4桁）
  - `mm`: 申請提出月（2桁ゼロ埋め）
  - `NNNN`: 同年月内の連番（4桁ゼロ埋め、例: `0001`）
- 差し戻しにより新規生成された申請も、新規提出時に新しい申請番号を採番する
- 元申請との紐付けは `parent_application_id` で管理する

**例**
```
2026-05-0001  （2026年5月 第1件目）
2026-05-0002  （差し戻し再提出、元は 2026-05-0001）
2026-06-0001  （2026年6月 第1件目）
```

> 連番はテナント内でリセットされる（異なるテナントが同じ申請番号を持つことは許容する）。

## 10. 用語定義

| 用語 | 定義 |
|---|---|
| **テナント** | システムを利用する1つの組織単位。データはテナント間で完全分離される |
| **システム管理者** | プラットフォーム全体を管理する特権ロール。テナントに属さず全テナントのデータを参照できる |
| **時間管理者** | テナント内の勤務時間・休暇を管理するロール。自テナントの申請全件の閲覧・編集・代行決裁・クローズ処理を担う |
| **クローズ処理** | 決裁完了後に時間管理者が内容確認を行い申請を「完了（closed）」状態に遷移させる操作 |
| **代行決裁** | 時間管理者が申請者または担当承認者に代わって承認・差し戻しを実施する操作（ログに代行フラグを記録） |
| **申請** | 決裁を求めて提出する行為・文書 |
| **合議** | 担当者がコメント・意見を付与する確認ステップ |
| **承認** | 直接の上位者が内容を認めるステップ |
| **決裁** | 最終権限者が正式に可否を決定するステップ |
| **差し戻し** | 処理者が申請者へ修正を求める操作。現申請をソフトデリートし、内容を複製した新規申請を自動生成してステップ1から再審査する |
| **取り消し** | 申請者・時間管理者・システム管理者が申請を無効化する操作。ソフトデリートされ復元不可（複製して新規申請は可能） |
| **決裁ルート** | 申請者ごとに設定された合議・承認・決裁の処理経路（クローズは全時間管理者が担当） |
| **年度** | 4月1日～翌年3月31日の期間（テナント単位のシステム設定で変更可） |
| **時間取得** | 休暇を15分単位の時間で申請する方式 |
| **日取得** | 休暇を1日単位で申請する方式 |
